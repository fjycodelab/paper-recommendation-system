# 架构概述

> 创建日期: 2026-05-24
> 状态: 初稿
> 来源: Vibe Coding V3 / 路径 A / A5 架构访谈

## 整体思路

系统拆成三条主线: Java 业务系统、Python 推荐推理服务、前端展示页面。

Java 业务系统负责登录认证、论文管理、用户行为记录、缓存读写、推荐结果聚合和对外 API。Python 推理服务负责把论文摘要、关键词、研究方向转成语义向量，并计算论文之间或用户偏好与论文之间的相似度。前端只做简洁展示: 登录、论文检索、论文详情、热门论文、用户偏好标签和推荐结果。

第一版先做真实但轻量的 embedding 相似度推荐，不做复杂训练平台。模型能力通过 FastAPI 服务隔离，后续可以替换成更复杂的召回、排序或在线学习模型，而不破坏 Java 业务层。

## 模块职责和边界

### 前端展示模块

职责:
- 登录页和基础会话状态展示。
- 论文列表、论文详情、热门论文、用户偏好标签和推荐结果展示。
- 搜索入口和下载入口展示。
- 收藏、评分、点击推荐、下载入口点击等行为触发。

边界:
- 不在前端实现推荐算法。
- 不做复杂状态管理，优先使用成熟 UI 组件减少代码量。
- 不直接访问数据库或 Python 推理服务，只调用 Java API。

暂定技术:
- Vue/Vite + Element Plus。真正创建前端代码前，再确认依赖。

### Java 业务服务

职责:
- 用户注册、登录、认证和基础权限控制。
- 论文信息、关键词、研究方向标签、论文来源链接和下载入口管理。
- 记录用户行为: 浏览、搜索、收藏、评分、推荐点击、下载入口点击等。
- 维护热门论文、用户偏好标签和推荐结果缓存。
- 通过 Spring WebClient 调用 FastAPI 推理接口。
- 聚合 MySQL、Redis 和 Python 推理结果，对前端返回统一 DTO。

边界:
- 不在 Java 内实现深度学习推理。
- 不直接耦合具体模型文件，只依赖稳定的推理接口协议。
- 爬虫能力先通过独立服务或任务接口隔离，避免污染核心业务。

### 数据访问与存储模块

职责:
- 使用 MyBatis-Plus 管理用户、论文、标签、行为、评分、收藏、推荐结果等表。
- MySQL 保存事实数据和可追溯行为数据。
- Redis 保存热点论文、用户短期行为、用户偏好标签和推荐结果缓存。

边界:
- MySQL 是最终一致的数据源。
- Redis 可丢失，不作为唯一事实来源。
- 临时行为可以先落 Redis，再由定时任务同步到 MySQL。

### Python 推理服务

职责:
- 加载轻量真实 embedding 模型。
- 接收论文摘要、关键词、研究方向，返回向量或相似度分数。
- 支持相似论文对比和用户偏好向量对论文集合的打分。

边界:
- 不处理用户登录和业务权限。
- 不直接写 MySQL 业务表。
- 对 Java 暴露稳定 HTTP API，模型内部可以替换。

第一版模型策略:
- 使用真实 embedding 模型做文本向量化。
- 输入文本由 `title + abstract + keywords + researchDirections` 拼接得到。
- 采用 cosine similarity 做相似度对比。
- 先保证推荐链路可信可运行，再扩展复杂排序模型。

### 爬虫与论文更新模块

职责:
- 支持论文搜索和下载入口。
- 支持按关键词、研究方向或来源配置定时更新论文元数据。
- 将抓取结果写入待审核或待入库流程，避免脏数据直接污染论文主表。

边界:
- 第一版可以先提供搜索/下载入口和简化抓取任务。
- 不把爬虫逻辑写死在 Controller 中。
- 后续按站点适配器扩展，不影响论文核心模型。

## 核心数据模型草案

- `user`: 用户账号、密码摘要、状态、创建时间。
- `paper`: 标题、作者、摘要、年份、来源、DOI/URL、下载入口、访问计数。
- `research_tag`: 研究方向标签。
- `paper_tag`: 论文与标签关联。
- `user_behavior`: 浏览、搜索、收藏、评分、点击、下载入口点击等行为流水。
- `favorite`: 用户收藏论文。
- `rating`: 用户评分论文。
- `paper_embedding`: 论文向量元数据、模型版本、向量存储引用或序列化向量。
- `recommendation_result`: 推荐批次、用户、论文、分数、推荐原因、模型版本。

## 主要用户流程

1. 用户登录。
2. 用户搜索论文或进入首页。
3. 系统展示热门论文、研究方向标签和个性化推荐。
4. 用户查看论文详情，触发浏览行为记录。
5. 用户收藏、评分、点击下载入口或点击推荐论文，系统记录行为。
6. 定时任务聚合用户行为，更新用户偏好标签和推荐缓存。
7. Java 服务调用 Python 推理服务，生成相似论文或个性化推荐分数。
8. 前端展示推荐结果和简单解释。

## 推荐入口假设

当前暂定 4 个推荐入口:

- 首页推荐: 展示热门论文和用户个性化推荐。
- 论文详情页: 展示与当前论文相似的论文。
- 搜索/下载入口: 根据搜索词和当前论文方向展示相关论文。
- 个人推荐页: 展示用户偏好标签和推荐列表。

如果后续你想收窄范围，优先保留“论文详情相似论文”和“首页热门论文 + 个性化推荐”。

## 关键技术决策

### 决策 1: Java 和 Python 解耦

理由:
- Java 更适合业务 CRUD、认证、缓存和接口聚合。
- Python 更适合 PyTorch 推理和 embedding 模型加载。
- HTTP 协议边界清晰，后续替换模型不影响业务层。

代价:
- 多一个服务需要启动和联调。
- 需要处理超时、失败和降级。

应对:
- Java 推荐接口设置超时和缓存兜底。
- Python 接口使用明确的请求/响应 DTO。

### 决策 2: Redis 只做缓存和临时行为缓冲

理由:
- 热门论文、推荐结果和用户偏好标签查询频繁，适合缓存。
- 用户行为可能高频写入，先写 Redis 再定时同步可减少 MySQL 压力。

代价:
- Redis 与 MySQL 会存在短暂不一致。

应对:
- 关键行为最终写入 MySQL。
- 缓存设置 TTL，并保留手动刷新能力。

### 决策 3: 第一版使用 embedding + cosine similarity

理由:
- 模型真实、简单、易解释，适合 MVP。
- 可以直接比较论文与论文、用户偏好与论文的语义相似度。
- 后续可升级为混合召回或排序模型。

代价:
- 推荐质量受摘要和关键词质量影响。
- 个性化程度不如复杂协同过滤或深度排序模型。

应对:
- 保存用户行为，为后续排序模型积累数据。
- 推荐结果带上标签和相似原因，便于人工观察效果。

## 预留扩展点

- 批量论文导入: 在论文入库前增加导入任务和审核状态。
- 模型评估: 增加推荐点击率、收藏率、评分反馈等指标。
- 推荐解释: 保存命中的关键词、研究方向和相似论文来源。
- 权限系统: 从普通用户扩展到管理员、数据维护者。
- 前端管理台: 在 API 稳定后增加论文管理、标签管理和推荐观察页面。
- 论文全文解析: 独立成文档解析模块，避免影响当前摘要推荐链路。
- 爬虫适配器: 每个来源独立适配，统一输出论文候选数据。

## 当前最大风险

1. **爬虫实时更新范围过大**: 不同站点规则、反爬和数据质量差异很大。第一版建议先做可配置搜索/下载入口和少量来源适配。
2. **用户行为“越多越好”容易失控**: 行为类型多会增加表设计和前端埋点复杂度。第一版先统一行为流水表，再逐步增加类型。
3. **模型依赖和本地环境不稳定**: PyTorch 和 embedding 模型可能受下载、显存、CPU 性能影响。第一版要保留模型版本、超时和缓存兜底。

## 待决策

- 第一版爬虫抓取来源: 先接哪个站点或公开数据入口。
- embedding 模型具体选择和本地缓存方式。
- 前端技术栈最终确认: 暂定 Vue/Vite + Element Plus。
- 用户登录是否需要 JWT 刷新机制，还是先做简单 token。
