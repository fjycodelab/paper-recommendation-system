# 项目叙事文件夹

这个目录用于维护论文推荐与科研行为分析系统的简历/面试叙事素材。

目标不是写流水账，而是把开发过程中的关键问题沉淀成可讲述的项目亮点: 为什么要做、遇到什么问题、用了什么技术、怎么解决、带来什么结果。

## 文件约定

- `narrative-log.md`: 项目叙事台账，持续追加和更新。
- `template.md`: 新叙事点模板。

## 叙事结构

每条叙事尽量按以下顺序写:

1. 场景: 当时要解决的业务或工程问题。
2. 问题: 具体卡点、风险或性能/维护性痛点。
3. 方案: 使用了哪些技术和设计。
4. 落地: 改了哪些模块或完成了哪些功能。
5. 验证: 测试、构建、接口、页面、日志或可量化指标。
6. 简历表达: 压缩成 1-2 条可放进简历的表述。
7. 面试追问: 面试官可能会问什么，应该怎么解释。

## 写作约束

- 不编造数字。没有测试过就写“待补指标”。
- 不只堆技术栈。每个技术必须服务于一个功能或问题。
- 不把计划当成果。未完成的内容只能写“计划/待实现”。
- 优先记录真实踩坑: Windows 路径、端口占用、Redis/MySQL 连接、模型加载、缓存一致性、搜索性能、下载失败等。
- 每个 spec 完成时至少补一条叙事。
- 每个明显 bug 或工程坑解决后，判断是否同时写入 `docs/lessons/`。

## 常用简历句式

- 在 [业务场景] 下，负责 [功能模块]，通过 [技术方案] 解决 [问题]，实现 [结果]。
- 针对 [痛点]，设计 [方案/流程]，将 [数据/请求/行为] 沉淀到 [存储/服务]，为 [后续能力] 提供基础。
- 为降低 [风险/耦合/查询压力]，将 [模块 A] 与 [模块 B] 解耦，并通过 [接口/缓存/异步/定时任务] 保证 [效果]。
- 在 [异常/边界场景] 中，通过 [测试/日志/降级策略] 定位并修复 [问题]，提升系统可维护性。
