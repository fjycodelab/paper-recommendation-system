# Spec 003: 用户收藏评分与行为记录

> 创建日期: 2026-05-26
> 状态: 已通过
> 对应 plan: 待创建 `docs/plans/003-user-behavior-plan.md`

## 目标

让系统能够记录用户围绕论文产生的核心行为，包括收藏、评分、详情浏览、搜索、筛选、外部链接点击和下载入口点击，并提供“我的收藏”“最近浏览”和管理员全局行为统计入口。

本 spec 完成后，系统不仅能展示论文，还能沉淀用户偏好和行为数据，为后续 Spec 004/005 的 embedding、个性化推荐、热门论文和偏好标签分析提供可追溯的数据基础。

## 用户故事

1. 作为普通用户，我想要收藏和取消收藏论文，以便后续快速找到自己关注的文献。
2. 作为普通用户，我想要给论文打 1-5 星评分，以便表达自己对论文价值或相关性的判断。
3. 作为普通用户，我想要查看我的收藏和最近浏览，以便继续阅读之前关注过的论文。
4. 作为系统，我想要隐式记录用户打开详情、搜索、筛选、点击外部 URL 和点击下载入口等行为，以便后续分析用户兴趣。
5. 作为管理员，我想要查看全局行为统计，以便了解论文访问、收藏、评分、下载入口点击等整体情况。
6. 作为后续推荐系统，我想要获得稳定的用户行为事件和显式偏好数据，以便计算热门论文、用户偏好标签和推荐解释。

## 功能需求

- [ ] 登录用户可以收藏一篇正常状态论文。
- [ ] 登录用户可以取消收藏已收藏论文。
- [ ] 收藏状态按 `user_id + paper_id` 唯一，一篇论文对同一用户只能有一个当前收藏状态。
- [ ] 登录用户可以对论文设置 1-5 星评分。
- [ ] 登录用户可以修改自己对论文的评分。
- [ ] 评分状态按 `user_id + paper_id` 唯一，同一用户对同一论文保留一个当前评分。
- [ ] 论文列表和详情需要展示当前用户是否已收藏，以及当前用户评分。
- [ ] 前端提供论文列表收藏按钮。
- [ ] 前端提供详情页评分入口。
- [ ] 前端提供“我的收藏”入口，展示当前用户收藏的正常状态论文。
- [ ] 前端提供“最近浏览”入口，展示当前用户最近打开过详情的论文。
- [ ] 系统隐式记录打开论文详情行为。
- [ ] 系统隐式记录搜索行为，字段只保留关键词、作者、年份。
- [ ] 系统隐式记录标签筛选行为。
- [ ] 系统隐式记录点击外部 URL 行为。
- [ ] 系统隐式记录点击下载入口行为。
- [ ] 同一用户短时间内重复打开、搜索或点击按多次行为记录，不做去重。
- [ ] 点击下载入口时既记录行为事件，也继续复用 Spec 002 的下载尝试链路记录下载结果。
- [ ] 隐式行为上报优先写入 Redis 行为队列，再异步批量落 MySQL。
- [ ] 显式收藏和评分直接写 MySQL，保证用户看到的当前状态可靠。
- [ ] 行为批量落库任务需要有互斥保护，避免多实例或重复调度导致同一批事件被并发处理。
- [ ] 管理员可以查看全局统计，包括行为事件总数、各事件类型数量、收藏总数、评分人数、平均评分、详情浏览较多论文、下载入口点击较多论文。
- [ ] 管理员统计第一版只做全局视角，不展示用户级明细。
- [ ] Java 服务继续作为唯一前端 API 入口；前端不直接访问 MySQL 或 Redis。
- [ ] 不新增独立 RabbitMQ/RocketMQ 依赖；第一版使用现有 Redis 能力实现轻量行为队列。若后续引入独立 MQ，需要另开 spec 或 ADR。

## 技术叙事要求

本 spec 的实现和后续项目叙事要能讲清楚“为什么用这些技术”，避免只堆技术名词。

- **Redis 解决的问题**: 详情浏览、搜索、筛选、点击等隐式行为频率高，但不需要同步阻塞用户主流程。先写 Redis 行为队列可以削平突发写入，避免每次前端交互都直连 MySQL。
- **消息队列解决的问题**: 行为上报和行为持久化异步解耦。前端交互只需要快速确认“事件已接收”，批量落库由后台任务处理，降低接口响应时间和数据库瞬时写压力。
- **锁解决的问题**: 行为批量落库任务可能被多实例或重复调度同时触发，需要用互斥锁保护消费过程，避免同一批事件被并发处理造成重复落库或统计异常。
- **MySQL 约束解决的问题**: 收藏和评分是用户可见的强状态，需要通过唯一键、事务或原子更新保证同一用户对同一论文只有一份当前状态。
- **不夸大指标**: 若没有压测或日志证据，不写“提升 xx%”“减少 xx ms”。可以描述“降低同步写库压力”“为后续推荐提供行为数据基础”，真实指标留到验收或压测后补充。

## 行为类型

第一版事件类型暂定:

- `PAPER_DETAIL_VIEW`: 打开论文详情。
- `PAPER_SEARCH`: 在论文工作台搜索，记录关键词、作者、年份。
- `TAG_FILTER`: 使用研究方向标签筛选。
- `EXTERNAL_URL_CLICK`: 点击论文外部 URL。
- `DOWNLOAD_CLICK`: 点击论文下载入口。

收藏和评分不只作为事件存在，还需要有当前状态表:

- `FAVORITE`: 收藏状态变化，可另写行为事件用于统计。
- `RATING`: 评分状态变化，可另写行为事件用于统计。

## 边界条件

- **未登录用户上报行为**: 返回 401，不记录匿名行为。
- **论文不存在**: 收藏、评分、打开详情、点击外部 URL、点击下载入口均返回明确错误，不写有效行为。
- **论文已软删除**: 普通用户不能收藏、评分或查看；行为记录应拒绝或不产生有效事件。
- **重复收藏**: 保持已收藏状态，不插入重复当前状态；是否记录一次重复点击事件由 plan 阶段决定。
- **重复取消收藏**: 保持未收藏状态，不报系统异常。
- **评分小于 1 或大于 5**: 返回明确参数错误。
- **重复评分**: 更新当前评分，并保留更新时间。
- **连续多次打开详情**: 按多次行为记录，不做去重。
- **搜索条件为空**: 不记录搜索行为，或记录为空搜索需在 plan 阶段明确；第一版倾向不记录空搜索。
- **搜索字段过长**: 返回参数错误或截断策略需在 plan 阶段明确，不能写入超长字段导致落库失败。
- **Redis 不可用**: 显式收藏和评分仍以 MySQL 为准；隐式行为记录可以失败但不能让论文浏览主流程崩溃，需要返回可观测错误或前端静默降级。
- **行为队列积压**: 后台批量落库应限制单批处理数量，并保留下一批继续处理。
- **批量落库部分失败**: 成功和失败需要可追踪，不能吞异常；失败事件是否回队列或记录失败日志由 plan 阶段确定。
- **多实例同时执行批量落库**: 必须通过互斥锁或等价机制避免并发消费同一批事件。
- **下载失败**: 下载失败不影响 `DOWNLOAD_CLICK` 行为记录；下载结果仍由 Spec 002 的下载尝试记录表达。

## 不在范围内

- 不做 embedding 生成、相似度计算或推荐排序；这些放到 Spec 004/005。
- 不做独立推荐解释字段；本 spec 只提供行为数据基础。
- 不做用户级管理员明细审计，只做全局统计。
- 不做反作弊、防刷、行为去重或风控策略。
- 不做精确一次消息投递保证；第一版接受至少一次或可追踪的批量落库语义。
- 不引入 RabbitMQ、RocketMQ、Kafka 等独立消息队列。
- 不做复杂实时大屏；管理员统计以基础表格和指标卡为主。
- 不做跨设备行为合并、匿名用户行为或隐私导出。
- 不改动 Spec 002 的论文主链路，除非为了展示收藏/评分状态做必要扩展。

## 验收标准

- [ ] 登录用户可以收藏论文，刷新后收藏状态仍然存在。
- [ ] 登录用户可以取消收藏论文，刷新后状态为未收藏。
- [ ] 登录用户可以设置并修改 1-5 星评分。
- [ ] 非法评分返回明确错误。
- [ ] 论文列表和详情能展示当前用户收藏状态和评分。
- [ ] “我的收藏”能展示当前用户收藏的正常状态论文。
- [ ] “最近浏览”能展示当前用户最近打开过详情的论文。
- [ ] 打开论文详情会产生 `PAPER_DETAIL_VIEW` 行为。
- [ ] 搜索会产生 `PAPER_SEARCH` 行为，并保存关键词、作者、年份。
- [ ] 标签筛选会产生 `TAG_FILTER` 行为。
- [ ] 点击外部 URL 会产生 `EXTERNAL_URL_CLICK` 行为。
- [ ] 点击下载入口会产生 `DOWNLOAD_CLICK` 行为，并继续保留下载尝试结果。
- [ ] 同一用户连续打开同一论文详情多次，MySQL 中能看到多条行为记录或统计计数增加多次。
- [ ] 隐式行为先进入 Redis 队列，再由后台任务批量落 MySQL。
- [ ] 批量落库任务具备互斥保护，测试覆盖重复调度或锁被占用场景。
- [ ] 管理员可以看到全局行为统计，不展示用户级明细。
- [ ] Java 相关测试通过。
- [ ] 前端构建通过。
- [ ] README 或模块文档补充 Spec 003 初始化、Redis 行为队列、行为落库和验收命令。
- [ ] `docs/narratives/narrative-log.md` 补充本 spec 的项目叙事，明确 Redis、消息队列语义、锁和 MySQL 约束分别解决的问题。

## 已决策

- 需要收藏和取消收藏。
- 需要 1-5 星评分。
- 搜索行为只保留关键词、作者、年份。
- 重复浏览、重复搜索和重复点击按多次记录。
- 管理端第一版只做全局统计。
- 需要使用 Redis 承载隐式行为队列或缓冲。
- 前端入口包括论文列表收藏按钮、详情页评分、我的收藏页、最近浏览页和行为统计面板。
- 本 spec 不做推荐、embedding 和热门论文排序。
- 项目叙事要偏向“用技术解决具体问题”，例如 Redis 削峰、消息队列异步解耦、锁避免并发消费、MySQL 约束保证收藏评分状态一致。

## 待决策

- 行为批量落库失败时，是重试、回队列、写失败表，还是仅记录错误日志。
- 最近浏览页按 MySQL 行为表查询，还是额外维护 Redis 最近浏览结构。
- 收藏和评分变化是否同时写入行为事件表，用于统计“收藏点击”和“评分点击”。
- 管理员全局统计是否需要单独聚合表，还是第一版实时 SQL 聚合。
- 隐式行为上报接口在 Redis 写入失败时返回错误，还是前端静默忽略。
