# A1 项目定义

> 创建日期: 2026-05-24
> 路径: Vibe Coding V3 / 路径 A / 阶段一

## 1. 这个项目解决什么问题?

为科研文献管理场景提供论文管理、用户科研行为沉淀、论文相似度计算和个性化推荐能力，帮助用户更快发现与自己研究方向相关的论文。

## 2. 谁会用? 什么场景用?

- 科研学习者: 收藏、评分、整理论文，并获得相关论文推荐。
- 研究人员或学生: 围绕研究方向维护标签，查看相似论文和热门论文。
- 系统管理员或维护者: 管理论文元数据、观察用户行为和推荐结果质量。

典型场景:
- 用户阅读论文后收藏或评分，系统沉淀偏好。
- 用户进入首页或推荐页，系统返回热门论文、相似论文和个性化推荐。
- 维护者批量补充论文摘要、关键词、研究方向标签。

## 3. MVP 是什么?

MVP 控制在 5 个能力内:

1. 论文基础信息与研究方向标签管理。
2. 用户收藏、评分、访问行为记录。
3. Python 推理服务生成论文语义向量和相似度分数。
4. Java 推荐接口聚合推荐分数、相似论文和用户偏好标签。
5. Redis 缓存热门论文、推荐结果和用户偏好标签。

## 4. 明确不做什么?

- 第一版不做复杂前端，优先保证后端 API 与算法服务闭环。
- 第一版不做真实大规模分布式训练，只做可运行的推理服务和可替换模型接口。
- 第一版不做在线学习或实时模型训练，用户行为先作为推荐输入数据沉淀。
- 第一版不做多租户、机构权限、论文版权全文解析。
- 第一版不做生产级监控告警，只保留日志和后续扩展口子。

## 5. 预期活多久?

按“几个月项目”处理: 流程完整执行，但实现范围以 MVP 为主，避免一开始铺太大。

## 待确认

- 是否需要前端页面，还是只交付后端 API 与接口文档。
- 论文数据来源: 手动录入、CSV 导入、公开数据集，还是后续接第三方接口。
- 推荐算法第一版是否允许使用轻量 mock embedding，再替换为真实 PyTorch 模型。
- 用户体系是否需要登录认证，还是先用简化用户表支撑行为记录。
