# ADR 002: Spec 002 使用 arXiv 元数据和本机下载尝试

> 日期: 2026-05-26
> 状态: 已接受，已实现
> 对应: Spec 002 论文信息与研究方向标签管理

## 背景

Spec 002 需要约 100 条真实样例论文，字段包含标题、作者、摘要、年份、来源、DOI、URL、下载链接、关键词、引用量和发布时间。后续 Spec 003-005 还会继续依赖这些论文数据做行为记录、embedding 和推荐。

样例数据必须公开、无需登录、可复现，同时不能把大体积 PDF 或有许可风险的文件提交进 git。下载入口也不能因为外部网络失败影响论文详情查看。

## 考虑过的选项

### 选项 1: 手写本地种子论文

**优点**:
- 完全离线可控。
- 不受外部网络和限流影响。

**缺点**:
- 数据不真实，摘要、作者、发布时间和来源质量低。
- 很难覆盖后续推荐和检索场景。

### 选项 2: 使用 arXiv Atom API 导入元数据

**优点**:
- 公开、无需登录，可拿到标题、作者、摘要、发布时间、分类、abs URL 和 PDF URL。
- arXiv 分类可以映射到固定二级研究方向标签。
- 通过 `source + source_paper_id` 可稳定去重。

**缺点**:
- 引用量通常缺失，需要允许为空。
- 依赖外部网络和 API 稳定性。
- PDF 下载要遵守个人研究用途和不再分发约束。

### 选项 3: 直接下载并纳入系统内文件分发

**优点**:
- 用户可以从系统内直接拿到 PDF。

**缺点**:
- PDF 文件大，不适合进入仓库。
- 存在版权和二次分发风险。
- 会把本 spec 从元数据管理扩展成文件服务，范围过大。

## 决策

选择 arXiv Atom API 作为 Spec 002 样例论文元数据来源；PDF 只做本机下载尝试和状态记录，不提交进 git，不在系统内对外二次分发。

## 理由

- Spec 002 的目标是建立论文元数据、标签、搜索和下载入口，而不是实现全文存储或文件分发平台。
- arXiv 元数据足够支撑真实列表、详情、搜索、标签映射和后续推荐样例。
- 下载结果记录 `SUCCESS`、`FAILED`、`TIMEOUT`、`NON_PDF`、`NO_URL` 等状态，可以为 Spec 003 行为记录预留入口。
- 下载目录放在 git 仓库之外，可以避免大文件、许可风险和仓库污染。
- Python 推理服务不参与 Spec 002，避免把 embedding 和推荐提前塞进论文管理闭环。

## 后果

**正面**:
- 本机 T10 已真实导入 arXiv 论文 100 条，满足样例规模。
- 本机 T10 已验证 1 次真实 PDF 下载成功和 1 次失败下载记录。
- 论文详情始终保留外部链接，即使下载失败也不影响用户查看来源。

**负面**:
- arXiv 网络失败时无法完成新增导入，需要记录失败原因或稍后重试。
- 本机下载路径只对当前机器有效，换机器需要重新配置 `APP_PAPER_DOWNLOAD_DIR`。
- 引用量字段可能为空，不能把引用量作为推荐或排序的强依赖。

**约束**:
- 导入接口请求体字段固定为 `query` 和 `maxResults`，一次上限 100。
- 下载文件默认保存到 `D:\code\codex_2\lencode\paper-downloads`，不进入 `javacode` 仓库。
- 仓库只保存代码、SQL、状态记录和文档，不保存下载到的 PDF。
- 后续如需全文解析、PDF 文件管理或系统内分发，必须另开 spec。

## 验证方式

- Java 测试通过: `Tests run: 95, Failures: 0, Errors: 0, Skipped: 0`。
- 前端 `npm run build` 通过。
- T10 API 级验收完成 arXiv 导入 `requested=100, imported=100, skipped=0, failed=0`。
- T10 下载成功记录包含文件名、大小和本机路径；失败记录包含状态和失败原因。
- MySQL 复核存在论文、标签和下载记录。
